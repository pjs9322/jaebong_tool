# Task 013: 편집 모드와 공유 모드 단일 구조 분리 작업

## 1. 개요 및 배경
현재 재봉툴 프로젝트의 아키텍처를 분석한 결과, `index.html` 뷰(View)와 `main.js` 컨트롤러 내부에서 `S.isViewerMode` 라는 분기 처리와 DOM 요소의 노출/비노출(Show/Hide)을 통해 두 가지 주요 모드(편집, 공유/열람)를 동시에 제어하고 있습니다.
사용자가 결과물을 외부인에게 '공유 링크'로 전달할 때 이 단일 파일 구조(SPA)는 여러 가지 심각한 불이익을 초래할 수 있어 이를 다음과 같이 진단하고 개선 태스크를 수립합니다.

## 2. 코드 분석 및 현행 구조의 위험성

### 1) Open Graph (og:title, og:image) 동적 렌더링의 구조적 한계 (가장 치명적)
* **현행 방식:** `index.html`의 `<head>` 영역에 메타 태그가 정적으로 박혀 있으며 ("홈페이지 유지보수 의뢰서를 생성해보세요"), 클라이언트 스크립트인 `core/viewer.js`의 `updateMetaTitle()` 함수를 통해서만 제목을 뷰어 모드에 맞게 동적 변경하고 있습니다.
* **불이익 원인:** 카카오톡, 슬랙, 인스타그램 등 대다수 메신저/SNS 링크 프리뷰 봇(Crawler)은 성능 및 보안 이슈로 자바스크립트를 아예 실행하지 않고, 서버에서 최초로 받아온 순수 HTML만을 파싱합니다.
* **결과:** 실제 수신자에게는 "수정 요청사항이 도착했습니다"라는 맞춤형 텍스트 대신, 무조건 "의뢰서를 생성해보세요(편집기 제목)"라는 잘못된 타이틀과 설명이 노출되어 서비스 신뢰도와 사용자 경험(UX)을 크게 감소시킵니다.

### 2) 클라이언트 성능 낭비 및 Over-fetching
* **현행 방식:** 공유 링크로 접속하는 단순 열람 사용자에게조차도 `index.html` 전체 문서 트리, 튜토리얼 마크업, 화면 캡처 관련 UI, 그리고 `main.js`가 임포트하는 캔버스 코어(`canvas.js`), 기능 생성(`history.js`) 등의 무거운 에셋이 전부 로드됩니다.
* **불이익 원인:** 뷰어 대상자(주로 모바일 기기 사용자, 클라이언트 및 외주 개발자 등)에게 불필요한 데이터를 다운로드 시키며, 로딩 시간과 브라우저 메모리 점유율이 심각하게 증가합니다. (단순한 JSON 데이터 기반 렌더링 페이지임에도 에디터를 로드하는 비용 지불)

### 3) 클라이언트 사이드 권한 제어의 보안 취약성
* **현행 방식:** `main.js` (라인 75~92)에서 API의 `owner_uuid`와 브라우저 로컬 `S.myUuid`를 자바스크립트로 비교한 뒤, 일치하지 않으면 뷰어 레이어를 켜고 편집기 레이어 버튼을 DOM에서 강제 숨김 처리(`display:none`)하고 있습니다.
* **불이익 원인:** 악의적인 사용자가 개발자 도구(F12)를 이용해 숨겨진 에디터 관련 DOM을 다시 노출시키거나, 로드되어 있는 함수(`S.history` 수정 및 `saveReqBtn` 등)를 호출하여 시스템을 교란할 수 있는 공격 표면(Attack Surface)을 엽니다.

### 4) 유지보수 및 코드 확장성 감소
* **현행 방식:** `index.html` 내에 편집기와 뷰어의 HTML이 혼재하며, CSS 또한 하나의 스코프에 속하게 됩니다.
* **불이익 원인:** 향후 에디터의 레이아웃을 크게 변경하거나 뷰어에만 특수한 채팅 UI(현재 Task 009로 비활성화 됨)를 재활성화 할 경우, 기존 로직과의 사이드 이펙트(CSS 충돌 등) 관리가 매우 까다로워집니다.

## 3. 해결 설계 및 목표

본 시스템 아키텍처를 보다 프로덕션 수준(Production-Level)으로 끌어올리기 위해 아래 솔루션 도입을 제안합니다.

1. **URL 구조 유지 및 PHP 기반 라우팅 처리 (핵심 요구사항 반영)**
   * 기존: 클라이언트 단(`index.html`)에서 자바스크립트로 GET 파라미터(`?id=`)를 파싱하여 조건부 UI 노출 처리.
   * 변경: 엔드포인트 URL 구조를 분기하지 않고 현재 상태를 유지하되, 진입점인 `index.php`에서 서버 사이드 분기 처리.
   * **편집 모드:** GET 파라미터(`id`)가 **없는** 경우(`/`). 편집기 앱 파일(`main.js`, `canvas.js` 등)과 UI를 서빙. (로컬 스토리지의 UUID를 통해 기존 작업 내역 동기화)
   * **공유 모드:** GET 파라미터(`id`)가 **있는** 경우(`/?id=xxxxx`). 뷰어 전용 에셋만 로드하고 편집기 관련 UI/JS는 아예 서빙하지 않음.

2. **SEO/Open Graph의 서버 사이드 렌더링 (SSR)**
   * 공유 모드 링크(`/?id=xxxxx`)로 접속 시, `index.php`에서 해당 `id` 데이터(수정 내역 등)를 DB에서 먼저 조회하여, `<head>` 태그 내 동적인 Open Graph 메타 태그(`og:title`, `og:image` 등)를 생성해 응답함.
   * 이를 통해 카카오톡, 슬랙 등 자바스크립트를 지원하지 않는 환경에서도 맞춤형 텍스트와 정보가 정상 노출됨.

3. **Viewer 전용 경량 자바스크립트/CSS 번들링**
   * 불필요한 Canvas/Capture 코드를 배제하고, 열람 전용인 `viewer.css` 및 `viewer-main.js` 만 다운로드.
   * `SyncAPI.readDocument` 호출(혹은 PHP가 직접 주입한 데이터) 후 단순 데이터 맵핑 및 HTML 렌더링만 진행하여 비용을 획기적으로 낮춤.

- [ ] **2. 편집 모드 URL에서 ID 파라미터 노출 제거:** `js/main.js`의 에디터 초기화 시 브라우저 주소창에 `history.replaceState`로 `?id=` 파라미터를 강제 추가하던 로직 제거. (편집자는 항상 `/` 루트 경로 유지)
- [ ] **3. 공유 모드 전용 뷰어 마크업 분리:** 파라미터가 있을 때 출력되는 뷰어 전용 HTML 마크업 구성. 편집용 UI(`<div id="editorLayer">`) 및 캔버스 요소 배제.
- [ ] **4. Viewer 전용 컨트롤러 스크립트 작성:** 기존 `js/core/viewer.js` 로직을 기반으로 캔버스 의존성을 제외한 `js/viewer-main.js`를 신규 제작하여 뷰어 모드에만 할당.
- [ ] **5. Open Graphic(OG) 동적 렌더링 연동:** 백엔드(PHP)에서 `$_GET['id']`를 통해 조회된 문서 데이터를 바탕으로 `og:title`, `og:description` 등을 삽입하는 로직 구현. (클라이언트 단의 `updateMetaTitle` 의존 로직 제거 혹은 병행 적용)
